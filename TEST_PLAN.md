# 統合テスト計画 — LINE Bot + 管理画面 MVP

## 1. テスト方針

### 対象バージョン
MVP段階3（STEP1〜5 実装完了）

### テスト種別
本ドキュメントは**手動統合テスト**の計画を扱う。  
単体テスト・E2Eテストフレームワークの導入はMVPスコープ外。

### テスト環境
| 項目 | 内容 |
|---|---|
| 実行環境 | `npm run dev`（ローカル）または Vercel プレビュー |
| DB | Supabase（開発プロジェクト） |
| LINE | LINE Developers コンソールのテスト用チャネル |
| Webhook 到達 | ngrok 等のトンネリングツールでローカルを公開 |

### 前提条件
- `.env.local` に全環境変数が設定済みであること
- Supabase の `faq` / `menus` / `conversations` テーブルが存在すること
- `LINE_OWNER_USER_ID` にオーナーの LINE ユーザーID が設定済みであること
- ngrok 等で `http://localhost:3000` を公開し、LINE Webhook URL に設定済みであること

## 2. テスト領域と優先度の考え方

| 領域 | 理由 |
|---|---|
| **高** | ユーザーへの直接影響・誤送信リスク・認証バイパスリスク |
| **中** | 機能正常系・UX に影響するが業務停止にはならない |
| **低** | エッジケース・UI の細部・将来的な回帰リスク |

## 4. テスト実行順序（推奨）

```
1. 環境準備（ngrok 起動・Webhook URL 設定）
2. C-01〜C-06（認証基盤の確認）
3. A-08（署名検証）
4. C-09〜C-11（FAQ 登録・編集・削除）
5. A-01〜A-06（FAQ 応答確認）← FAQが登録された状態で実施
6. A-07（応答時間確認）
7. B-01〜B-05（エスカレーション通知）
8. C-15〜C-18（会話ログ確認）← 上記で会話が蓄積された状態で実施
9. D-01〜D-07（お知らせ配信）
10. C-07（ダッシュボード統計）← データが揃った状態で確認
11. C-20〜C-25（メニュー管理）
12. E-01〜E-05（エッジケース）
```

## 6. 既知の制限事項（テスト対象外）

- Claude API のレスポンス品質（どの確信度になるかはモデル依存）
- LINE Messaging API の配信遅延（LINEサーバー側の問題）
- Supabase の書き込みレイテンシ
- メニュー管理の `duration` / `category` / `is_active`（スコープ外）
